# Docker Volumes

Docker volumes persist data generated by a container outside of the container's own filesystem, so the data survives container removal. This walkthrough uses a MongoDB container to first show the data-loss problem, then fixes it with a named volume.

## Part 1: Without a volume — data is lost

1. Create a container, mapping host port 27018 to container port 27017:

   ```sh
   docker run -d -p 27018:27017 mongo
   ```

   - `docker run` — create and start a new container.
   - `-d` — detached mode; run the container in the background and print its ID.
   - `-p 27018:27017` — publish a port, mapping host port `27018` to the container's port `27017` (MongoDB's default port), so `localhost:27018` on the host reaches Mongo inside the container.
   - `mongo` — the image to run (pulled from Docker Hub if not already local).
   - No `-v` is given, so Mongo's data directory (`/data/db`) lives only inside this container's writable filesystem — it disappears with the container.

2. List the container:

   ```sh
   docker ps
   ```

3. Connect to MongoDB with Compass using the connection string:

   ```text
   mongodb://localhost:27018/
   ```

4. Add data — click **Create Database**, set Database Name `Atul` and Collection Name `New`, then open `Atul.New` and **Add Data → Insert Document**:

   ```json
   {
     "name": "Atul"
   }
   ```

5. Stop the container:

   ```sh
   docker stop 9d04c9431cd2
   ```

6. Verify the data is **not** retained — start a fresh container on the same port and reconnect:

   ```sh
   docker run -d -p 27018:27017 mongo
   docker ps
   ```

   Reconnecting Compass to `mongodb://localhost:27018/` now shows no `Atul` database. The previous container wrote MongoDB's data to its own filesystem at `/data/db`; once that container was stopped and replaced, that filesystem — and the data on it — was gone.

## Part 2: With a volume — data is retained

7. Create a named volume for MongoDB's data:

   ```sh
   docker volume create mongodata
   docker volume ls
   ```

8. Stop the unmounted container from Part 1, then create a new container with the volume attached at Mongo's data directory (`/data/db`):

   ```sh
   docker stop 923c90d0002f
   docker run -d -p 27018:27017 -v mongodata:/data/db mongo
   docker ps
   ```

   - `docker run -d` — same as before: create and start a container in the background.
   - `-p 27018:27017` — same port mapping as before.
   - `-v mongodata:/data/db` — mount the named volume `mongodata` (created in step 7) at `/data/db` inside the container, which is exactly where MongoDB stores its data. Anything Mongo writes now goes into the volume, not the container's filesystem.
   - `mongo` — the image to run.

9. Connect to MongoDB with Compass again at `mongodb://localhost:27018/` and add the same data as before (`Atul.New` → `{ "name": "Atul" }`).

10. Stop this container too:

    ```sh
    docker stop 900e72dfa901
    ```

11. Create another new container, mounting the **same** `mongodata` volume:

    ```sh
    docker run -d -p 27018:27017 -v mongodata:/data/db mongo
    docker ps
    ```

12. Verify the data **is** retained — reconnect Compass to `mongodb://localhost:27018/`. The `Atul` database, `New` collection, and the `{"name": "Atul"}` document are still there, because both containers mounted the same `mongodata` volume at `/data/db` — the data lived in the volume, not in either container.

## Part 3: Bind mount — share a host folder with a container

Parts 1 and 2 used a **named volume** (`mongodata`), which Docker creates and manages under its own storage area. A **bind mount** is the other way to attach storage to a container: instead of a volume name, you give an actual path on the host, and that host folder is mapped directly into the container.

13. Run an nginx container, bind-mounting the host folder `C:/mongodb` to `/data` inside the container:

    ```sh
    docker run -d -p 8080:80 -v C:/mongodb:/data nginx
    ```

    - `docker run -d` — create and start the container in the background.
    - `-p 8080:80` — publish host port `8080` to nginx's port `80` inside the container.
    - `-v C:/mongodb:/data` — bind mount: because the left side (`C:/mongodb`) is a host path rather than a volume name, Docker mounts that exact host folder at `/data` in the container, instead of mounting a Docker-managed volume.
    - `nginx` — the image to run.

16. Open a shell inside the container (Docker Desktop **Exec** tab, or `docker exec -it <container_id> sh`) and check the mounted folder:

    ```sh
    ls
    cd data
    ls
    ```

    The file created on the host (`Atul.txt`) shows up inside the container at `/data` immediately — no rebuild or restart needed.

17. The reverse is also true: any file the container writes into `/data` appears in `C:\mongodb` on the host. This live, two-way sync is what makes bind mounts useful for things like serving static files or live-editing code from the host while a container runs — unlike a named volume, where the data only lives inside Docker's managed storage and isn't directly browsable on the host filesystem.

## Useful inspection commands

```sh
docker volume inspect mongodata
docker volume rm mongodata
```
